# Context Schema

## Required Files

- `README.md`: describe how the project uses `.context/`
- `INDEX.md`: generated shard index with latest active shard and all log shards
- `vision.md`: product intent, user reality, root cause, measurable success, non-goals, hard FAQ
- `master-plan.md`: goals, phases, milestones, acceptance criteria, non-goals
- `design.md`: problem, domain model, change axes, interface contracts, technical choices, NFRs, Walking Skeleton, risk order
- `current-status.md`: current phase, done, in progress, next, blockers, active work-log shard
- `task-breakdown.md`: concrete task cards and priority
- `work-log.md`: append-only structured execution history
- `bug-log.md`: engineering problems, root causes, solutions, preventions
- `decisions.md`: structured architecture decision records
- `risk-register.md`: risks, triggers, mitigations, de-risk order

## Design Gate

`validate_context.py` is a hard gate. It fails when required files are missing, required fields are absent, or design documents still contain `[待更新]`, `[待填]`, or `TODO`.

### vision.md

Required sections:

- 一句话愿景
- 目标用户与他们的真实处境
- 表层诉求 vs 根因
- 解法与差异点
- 成功定义
- 非目标 / 明确不做
- 硬核 FAQ

### design.md

Required markers checked by validation:

- 领域模型
- 可预见的变化轴
- 接口契约
- 选型

Required sections:

- `0 问题一句话`
- `1 领域模型（核心实体与关系）`
- `2 可预见的变化轴`
- `3 模块拆分与接口契约`
- `4 技术选型`
- `5 非功能需求（NFR）`
- `6 Walking Skeleton`
- `7 最高风险与降险顺序（De-risk first）`

The change-axis table must use:

| 维度 | 当前形态 | 未来可能 | 本次是否预留 | 理由 |
| --- | --- | --- | --- | --- |

Only reserve extension points for changes marked as certain. Everything else is YAGNI.

### master-plan.md

Must include `非目标 / 明确不做`.

### current-status.md

Required fields:

- 当前阶段
- 已完成
- 进行中
- 下一步
- 阻塞项
- 当前活跃日志分片

`当前活跃日志分片` must point to the highest-numbered existing `work-log` shard.

### INDEX.md

Generated by `init_context.py`, `append_work_log.py`, `record_bug.py`, and `record_decision.py`.

Required content:

- latest active shard for `work-log`
- latest active shard for `bug-log`
- latest active shard for `decisions`
- all existing shards for each rotating log

Validation fails if `INDEX.md` is missing, contains placeholders, or omits an active/existing shard.

## Structured Entry Requirements

### work-log.md

- 时间
- 目标
- 动作
- 结果
- 验证
- 下一步

### bug-log.md

- 标题
- 现象
- 触发条件
- 影响
- 根因
- 解决方案
- 预防措施
- 状态

### decisions.md

ADR entries are append-only and require:

- 编号
- 标题
- 状态：`proposed`、`accepted`、`superseded`
- 日期
- 背景（Context）
- 决策（Decision）
- 取舍（利弊，强制写明放弃了什么）
- 后果（Consequences）
- 取代关系（Supersedes）

The `取舍` field is mandatory. A decision that does not state what was given up is not a decision record.

## Rotating Logs

Rotating logs:

- `work-log.md`
- `bug-log.md`
- `decisions.md`

Constant:

- `MAX_LOG_LINES = 1500`

Naming:

- Base file is shard 1: `work-log.md`
- Later shards use forward numbering: `work-log-2.md`, `work-log-3.md`, and so on
- The current active shard is the highest-numbered existing shard

Write rules:

- If no shard exists, write to the base file.
- If the active shard has fewer than 1500 lines, append there.
- If the active shard has at least 1500 lines, create the next shard and append the whole record there.
- Rotation happens only between records; a single record is never split.
- Non-first shards start with a navigation header: `本文件为 <base> 第 N 片，续自第 N-1 片`.
- Old shards are read-only history.
- `INDEX.md` is refreshed after each append.

Line counting uses `content.count("\n") + 1`; empty files count as 0 lines.
